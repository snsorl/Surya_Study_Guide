# Prompt Logs

## Learning Objectives
- Explain the role of a Prompt Log in developing metacognitive programming habits.
- Structure prompt log entries using the standard 5-part format: Context, Prompt, Output, Evaluation, and Revisions.
- Critically evaluate AI-generated code snippets for security, efficiency, and edge case issues.
- Maintain a record of prompt iterations to improve prompt engineering skills over time.

---

## Why This Matters
When developers start using AI coding assistants, they often fall into a bad habit of trial-and-error:
1.  They type a short, vague prompt.
2.  The AI outputs broken or incomplete code.
3.  The developer copies it, sees an error, tweaks their prompt randomly, and tries again.

This trial-and-error cycle is a massive waste of time and prevents you from actually learning *why* the code failed. To break this cycle, professional developers maintain a **Prompt Log**. A prompt log is a developer journal where you record your prompts, the code you received, and your critical evaluation of that code. Keeping a log turns AI usage from a random guessing game into a structured, scientific engineering workflow.

---

## The Concept

### What is a Prompt Log?
A Prompt Log is a markdown journal kept in your project repository (often inside a `docs/` or `logs/` directory). By forcing yourself to write down a critique of every AI suggestion, you train your brain to read code skeptically, rather than blindly accepting completions.

---

### The Standard 5-Part Log Format
Every entry in your Prompt Log must follow this layout:

#### 1. Context
Describe the specific feature or bug fix you are trying to implement. Mention any constraints (e.g. Java version, variable structures).
> *Example*: *"Creating a utility to calculate the average of an integer array."*

#### 2. Prompt
Paste the exact text prompt you sent to the AI assistant.
> *Example*: *"Write a Java method that calculates the average of an int array."*

#### 3. Output
Insert the code block generated by the AI inside standard markdown code blocks.

#### 4. Evaluation (Critique)
This is the most critical section. Write a bullet-point analysis of the AI's code. You must check:
*   **Compilation**: Does it compile without warnings?
*   **Null Safety**: What if references passed are `null`?
*   **Boundary Conditions**: What if arrays are empty, numbers are zero, or indexes are out of bounds?
*   **Security**: Are there hardcoded secrets or raw queries?
*   **Performance**: Does the loop run in efficient time (e.g. `O(N)`)?

#### 5. Revised Prompt (The Correction)
If the evaluation reveals flaws, document the follow-up prompt you sent to instruct the AI to fix them. If multiple turns were required, record them sequentially.

## Code Example: Walkthrough of a Prompt Log Entry
Let's see an actual prompt log entry demonstrating this workflow.

### 1. Context
We need a method that calculates the average value of an array of integers. The result should be returned as a double to preserve decimal accuracy.

### 2. Prompt
> *"Write a static Java method named findAverage that takes an int[] array and returns the average as a double."*

### 3. Output
```java
public static double findAverage(int[] arr) {
    int sum = 0;
    for (int i = 0; i < arr.length; i++) {
        sum += arr[i];
    }
    return sum / arr.length;
}
```

### 4. Evaluation
- **Syntax**: Compiles successfully.
- **Precision Bug**: `sum / arr.length` performs **integer division** first, throwing away the fractional part before casting the final result to a double! For example, if sum is 5 and length is 2, `5 / 2` evaluates to `2` (integer), which then returns `2.0` instead of `2.5`.
- **Null Safety Failure**: If the caller passes `null` for `arr`, the JVM crashes with a `NullPointerException` when evaluating `arr.length` in the loop.
- **Division-by-Zero Risk**: If the caller passes an empty array `new int[0]`, the code evaluates `sum / 0`, throwing an `ArithmeticException` and crashing the program.

### 5. Revised Prompt
> *"The method has three issues:*
> *1) It performs integer division (sum / arr.length) which loses decimal precision. Cast the sum to double first.*
> *2) It crashes with a NullPointerException if the array is null.*
> *3) It crashes with an ArithmeticException if the array is empty.*
> *Please add guard checks to return 0.0 if the array is null or empty, and fix the decimal division."*

### Final Safe Code Suggestion
The AI processes the feedback and produces the corrected, secure method:

```java
public class ArrayMath {

    /**
     * Calculates the average of an array of integers.
     * Safely returns 0.0 if the array is null or empty, and preserves decimal precision.
     */
    public static double findAverage(int[] arr) {
        // 1. Guard Check: Handle null or empty arrays safely
        if (arr == null || arr.length == 0) {
            System.out.println("Warning: Input array is null or empty. Returning 0.0");
            return 0.0;
        }

        // Use long for sum to prevent integer overflow on large arrays
        long sum = 0; 
        for (int value : arr) {
            sum += value;
        }

        // 2. Cast sum to double to ensure double division is executed
        return (double) sum / arr.length;
    }

    public static void main(String[] args) {
        // Verification checks
        int[] scores = {5, 6};
        System.out.println("Average: " + findAverage(scores)); // Prints 5.5 (Correct!)

        System.out.println("Empty check: " + findAverage(new int[0])); // Prints 0.0
        System.out.println("Null check: " + findAverage(null)); // Prints 0.0
    }
}
```

## Summary
- A **Prompt Log** is a technical journal tracking your interactions with AI assistants.
- Keeping logs develops **metacognition**—forcing you to slow down and critique code rather than clicking accept blindly.
- The standard entry consists of **Context, Prompt, Output, Evaluation, and Revisions**.
- Critical evaluations focus on compilation warnings, null safety, boundary values, security, and precision errors.
