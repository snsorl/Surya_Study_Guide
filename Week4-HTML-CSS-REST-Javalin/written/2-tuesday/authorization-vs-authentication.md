# Authentication vs. Authorization

## Learning Objectives
- Define the fundamental difference between Authentication and Authorization.
- Identify common authentication schemes: Basic Authentication, API Keys, and Bearer Tokens.
- Implement an authentication check filter in Javalin using a `before()` route handler.
- Respond with appropriate HTTP status codes (401 and 403) for security failures.

---

## Why This Matters
When exposing REST APIs to the public web, security is not an optional add-on. If you fail to protect your endpoints, malicious actors can write script engines to query databases, delete records, or extract sensitive customer information. To protect your application, you must enforce a digital gatekeeper. First, you must prove who a user is (Authentication), and second, you must verify if they are allowed to perform the requested action (Authorization). Implementing these checks at the middleware layer ensures your endpoints remain secure by default.

---

## The Concept

### The Security Definitions
While often used interchangeably, authentication and authorization are distinct security steps:

-   **Authentication (AuthN)**: The process of verifying *who* a user or client is. The user provides credentials (like a username/password or token) to prove their identity.
    -   *Question*: "Who are you?"
    -   *Result*: A verified Identity (e.g., "This is User 42").
    -   *HTTP Failure Code*: **401 Unauthorized** (confusingly named; it actually means *unauthenticated*).
-   **Authorization (AuthZ)**: The process of verifying *what* an authenticated user is allowed to do. Once we know who the user is, we check their permissions, groups, or roles before executing the requested action.
    -   *Question*: "Are you allowed to do this?"
    -   *Result*: Allow or Deny (e.g., "User 42 is a 'User', not an 'Admin', so they cannot delete this resource").
    -   *HTTP Failure Code*: **403 Forbidden**.

### Common API Authentication Schemes

#### 1. Basic Authentication
The client sends the username and password encoded in Base64 within the `Authorization` header.
-   *Format*: `Authorization: Basic <Base64-encoded credentials>`
-   *Example*: `Authorization: Basic am9obmRvZTpwYXNzd29yZDEyMw==`
-   *Note*: Base64 is not encryption; it is simply an encoding scheme. Basic Auth must always run over HTTPS to prevent credential theft.

#### 2. API Keys
A simple, static secret string generated by the server and issued to the client. The client passes this string either in a custom header (e.g., `X-API-Key`) or as a query parameter.
-   *Format*: `X-API-Key: my-secret-api-key-value`
-   *Use Case*: Commonly used for system-to-system integrations or public developer APIs.

#### 3. Bearer Tokens
A cryptographic token (like a JSON Web Token) that "bears" authentication claims. The client obtains the token after completing a login process and attaches it to subsequent headers.
-   *Format*: `Authorization: Bearer <token>`
-   *Use Case*: Standard for modern single-page applications (SPAs) and mobile apps consuming REST APIs.

### Intercepting Requests in Javalin
Instead of duplicating authentication check logic inside every route controller, Javalin allows you to implement a global interceptor using `app.before()`. This interceptor runs before matching endpoints. If a request fails the security check, the handler can throw an exception or set a status code and halt execution immediately.

## Code Example

Here is an example showing how to configure basic API Key authentication using a Javalin `before()` filter.

```java
import io.javalin.Javalin;
import io.javalin.http.UnauthorizedResponse;
import io.javalin.http.ForbiddenResponse;

public class AuthApp {

    // Simple mock database of valid API Keys and user roles
    static String getRoleFromApiKey(String apiKey) {
        if ("super-secret-admin-key".equals(apiKey)) {
            return "ADMIN";
        } else if ("user-secret-key".equals(apiKey)) {
            return "USER";
        }
        return null; // Key invalid
    }

    public static void main(String[] args) {
        Javalin app = Javalin.create().start(8080);

        // Global Before Filter for all /api/ endpoints
        app.before("/api/*", ctx -> {
            // Extract the API Key from header
            String apiKey = ctx.header("X-API-Key");

            if (apiKey == null || apiKey.isBlank()) {
                // Halt execution and return 401 if key is missing
                throw new UnauthorizedResponse("API Key is missing in request headers.");
            }

            String role = getRoleFromApiKey(apiKey);
            if (role == null) {
                // Halt execution and return 401 if key is invalid
                throw new UnauthorizedResponse("Invalid API Key provided.");
            }

            // Save user role in request attribute context for downstream handlers
            ctx.attribute("userRole", role);
        });

        // Endpoint accessible to any authenticated user
        app.get("/api/data", ctx -> {
            String role = ctx.attribute("userRole");
            ctx.result("Here is basic API data. Access granted for role: " + role);
        });

        // Admin-only Endpoint
        app.delete("/api/data", ctx -> {
            String role = ctx.attribute("userRole");
            
            // Check Authorization
            if (!"ADMIN".equals(role)) {
                // Return 403 if authenticated but lacking correct privileges
                throw new ForbiddenResponse("Access Denied: Admin role required.");
            }
            
            ctx.result("Resource successfully deleted by Admin.");
        });
    }
}
```

---

## Summary
-   **Authentication (AuthN)** verifies identity ("who you are"), returning **401 Unauthorized** on failure.
-   **Authorization (AuthZ)** verifies permissions ("what you can do"), returning **403 Forbidden** on failure.
-   Standard authentication headers include **Basic Auth**, **API Keys**, and **Bearer Tokens**.
-   Implement security filters globally in Javalin using **`app.before()`** to dry up controller code and secure routes by default.
