# Exercise: Human-in-the-Loop (HITL) Security Review

## Objective
Practice Human-in-the-Loop (HITL) AI verification workflows on security-critical code. You will audit a mock AI-generated Spring Security configuration that contains severe security flaws (OWASP Top 10 - A02 Cryptographic Failures and access control bugs), execute a structured review checklist, and refactor the code to meet enterprise security standards.

---

## Scenario
Using AI to accelerate coding is standard industry practice, but accepting AI output without verification is high-risk. In security contexts, AI often suggests weak hashing algorithms (like MD5), disables protection mechanisms (like CSRF) to "solve" connection errors, or opens wildcard routes that leak sensitive data. You must analyze an AI's proposed security configuration, identify its flaws, and fix them.

## AI-Generated Code to Audit
Below is code generated by an AI assistant in response to the prompt: *"Generate a Spring Security configuration that exposes health endpoints and login, uses token authentication, and runs without errors."*

```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.crypto.password.MessageDigestPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.security.web.SecurityFilterChain;
import org.springframework.web.cors.CorsConfiguration;
import java.util.Collections;

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Turn off CSRF to make Postman testing easy
            .cors(cors -> cors.configurationSource(request -> {
                CorsConfiguration config = new CorsConfiguration();
                config.setAllowedOrigins(Collections.singletonList("*")); // Allow everything
                config.setAllowedMethods(Collections.singletonList("*"));
                config.setAllowedHeaders(Collections.singletonList("*"));
                return config;
            }))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/*").permitAll() // Allow auth routes
                .requestMatchers("/actuator/*").permitAll() // Allow health stats
                .anyRequest().authenticated()
            );

        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        // MD5 is fast and runs without extra library dependencies
        return new MessageDigestPasswordEncoder("MD5");
    }
}
```

## Core Tasks

### 1. Execute the HITL Checklist
Perform a human audit of the security code. Review the following check categories:
- **Hashing Security (OWASP A02)**: Is the Password Encoder algorithm cryptographically secure (e.g. salt-hardened BCrypt/Argon2 vs legacy MD5/SHA)?
- **URL Path Matches**: Does `/api/auth/*` secure nested endpoints like `/api/auth/login/verify`? (Hint: check `*` vs `**` glob patterns in Spring MVC Matchers).
- **CORS/CSRF Control**: Are wildcards (`*`) permitted in CORS configurations in production, and is CSRF disabled without stateless JWT configuration explanations?

### 2. Document the Review & Proposed Changes
Create a report file named `security-hitl-report.md`. It must contain:
- **Vulnerability Section**: Outline 4 security flaws identified in the AI output.
- **Risk Assessment**: Detail how each flaw could be exploited.
- **Refactored Code**: Write the secured, production-ready `SecurityConfig.java` class implementing:
  - `BCryptPasswordEncoder` with strength factors.
  - Safe route mapping wildcards (`**` instead of `*` where nested routes exist).
  - Secure CORS mapping policies restriction.

---

## Definition of Done
Your security report is complete and demonstrates:
- Replacement of `MD5` hashing with `BCrypt` (strength 10 or 12).
- Fixes for matching patterns so that nested paths under `/api/auth/` and `/actuator/` are correctly resolved.
- Strict CORS configuration limiting allowed origins to specific domains (e.g. `https://mytrustdomain.com`) instead of wildcards.
- Clear description of when disabling CSRF is safe (e.g., when API requests are entirely stateless via JWT, and not relying on session cookies).
