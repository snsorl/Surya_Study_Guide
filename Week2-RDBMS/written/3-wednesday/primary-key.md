# Primary Keys

## Learning Objectives
- Define the purpose and properties of a Primary Key (PK) in relational tables.
- Compare Natural Keys and Surrogate Keys, highlighting their architectural trade-offs.
- Explain the relationship between Primary Key constraints and the implicit NOT NULL and UNIQUE requirements.
- Implement primary keys in SQL using standard and auto-incrementing data types.

---

## Why This Matters
In a relational database, duplicate records are the enemy of data integrity. If a table allows two rows to be completely identical, or allows two separate customers to share the exact same ID number, your data becomes corrupt. How would the application know which customer to charge? How would a shipping system determine which address to ship to?

To prevent this, every relational table must have a way to uniquely identify each row. This is the role of the **Primary Key**. 

Selecting the right column (or columns) to act as your primary key is a foundational database design decision. Choosing poorly (such as using a user's phone number as a primary key, which can change or be shared) can lead to massive data migration headaches and application bugs later.

---

## The Concept

### 1. What is a Primary Key?
A **Primary Key** is a column (or group of columns) that uniquely identifies each row in a table. 

To act as a primary key, the selected column must adhere to three strict rules enforced by the RDBMS engine:
1.  **Uniqueness:** No two rows in the table can have the same primary key value.
2.  **Non-Nullability:** The primary key column cannot contain a `NULL` value. Every row *must* have an identity.
3.  **Immutability (Best Practice):** Once assigned, the primary key value should never change.

When you declare a column as a `PRIMARY KEY`, the RDBMS automatically creates a unique index on that column to optimize search speeds.

### 2. Natural Keys vs. Surrogate Keys
When designing a table, you must choose between two categories of primary keys:

#### A. Natural Keys
A primary key that is composed of attributes that already exist in the real world and are part of the entity's natural data.
-   *Examples:* Social Security Number (SSN), Book ISBN, Vehicle Identification Number (VIN), or email address.
-   *Pros:* Real-world meaning; prevents duplicate logic because the natural attribute is already unique.
-   *Cons:* Real-world rules change. If a government changes its SSN structure, or a user wants to change their email, updating a natural primary key can break database relationships across multiple tables.

#### B. Surrogate Keys
A primary key that has no real-world meaning and is artificially generated by the database engine solely for identity tracking.
-   *Examples:* Auto-incrementing integers (`id`) or Universally Unique Identifiers (UUIDs).
-   *Pros:* Completely stable, immutable, and small in size. Real-world business rule changes have zero impact on the database structure.
-   *Cons:* Allows duplicate real-world data unless you add a separate `UNIQUE` constraint to the natural columns (e.g., preventing two customer accounts with the same email).

**Enterprise Standard:** Surrogate keys are highly preferred in modern full-stack systems.

## Code Example

Let's look at how to declare primary keys in PostgreSQL. We will create two tables: one using an auto-incrementing surrogate key, and one using a natural key.

### Example 1: Surrogate Key using IDENTITY
In modern PostgreSQL, the standard way to create an auto-incrementing integer key is using `GENERATED BY DEFAULT AS IDENTITY`.

```sql
CREATE TABLE employees (
    -- Surrogate key: Auto-generates integers starting at 1
    employee_id INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    first_name VARCHAR(50) NOT NULL,
    last_name VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL  -- Natural identifier secured with UNIQUE
);

-- When inserting, we do not provide a value for employee_id
INSERT INTO employees (first_name, last_name, email)
VALUES ('Jane', 'Doe', 'jane.doe@email.com');

-- Resulting row: (1, 'Jane', 'Doe', 'jane.doe@email.com')
```

---

### Example 2: Natural Key
Suppose we are cataloging books. We might use the ISBN (International Standard Book Number) as a natural key.

```sql
CREATE TABLE books (
    -- Natural key: Must be provided manually during insert
    isbn CHAR(13) PRIMARY KEY,
    title VARCHAR(150) NOT NULL,
    author VARCHAR(100) NOT NULL
);

INSERT INTO books (isbn, title, author)
VALUES ('9780132350884', 'Clean Code', 'Robert C. Martin');
```

If you try to insert another book with the same ISBN, the engine throws a primary key violation error and rejects the write.

---

## Summary
-   A **Primary Key (PK)** is the unique identifier for a table row, implicitly enforcing `NOT NULL` and `UNIQUE` constraints.
-   **Natural Keys** are real-world identifiers (like ISBN). They carry risk because real-world values can change.
-   **Surrogate Keys** are database-generated placeholders (like auto-incrementing integers or UUIDs) and are the industry standard for database design stability.
-   Declare surrogate keys in PostgreSQL using the **`GENERATED AS IDENTITY`** clause.
