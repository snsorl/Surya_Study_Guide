# Auto-Incrementing Fields and Identity Sequences

## Learning Objectives
- Compare the legacy `SERIAL` data type and the modern `GENERATED AS IDENTITY` constraint in PostgreSQL.
- Explain the role of database Sequence objects in backing auto-increment columns.
- Construct SQL statements to inspect, modify, and reset identity sequences.
- Formulate database schemas using standard SQL-compliant identity definitions.

---

## Why This Matters
On Wednesday, we learned that surrogate keys (such as database-generated integers) are the industry standard for table primary keys. They provide a stable, immutable identity that protects table integrity.

However, how does the database know what number to assign next when you insert a new record? Behind the scenes, the database runs a specialized counter called a **Sequence**. 

If you do not understand how sequences operate, you will struggle with database migration issues. For example, if you manually copy data from one table to another (inserting specific ID values), the underlying sequence can fall out of sync. The next time your Java application tries to insert a record, the database will try to assign a duplicate ID that already exists, throwing primary key violation exceptions. Knowing how sequences work and how to reset them is key to maintaining data sync.

---

## The Concept

### 1. Legacy SERIAL vs. Modern IDENTITY
PostgreSQL has historically used two different syntaxes to generate auto-incrementing numbers:

#### A. Legacy: SERIAL Data Types
The `SERIAL` type is a syntax shortcut. When you declare a column as `SERIAL`, PostgreSQL automatically:
1.  Creates a separate Sequence object.
2.  Sets the column's default value to retrieve the next number from that sequence (`nextval()`).
3.  Establishes the sequence owner as the column, so if you drop the table, the sequence is deleted.

-   *Example:* `user_id SERIAL PRIMARY KEY`
-   *The Problem:* `SERIAL` is a proprietary PostgreSQL extension, not ANSI-SQL standard. Furthermore, it does not prevent users from manually inserting values into the column, which easily throws sequences out of sync.

#### B. Modern Standard: GENERATED AS IDENTITY (SQL:2003 Standard)
Introduced in PostgreSQL 10, this is the modern, SQL-compliant standard. It uses explicit identity constraints.
-   *Example:* `user_id INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
-   *Key Options:*
    -   `GENERATED BY DEFAULT AS IDENTITY`: The database auto-generates the number, but allows the user to override it by supplying their own ID if needed. (Standard choice).
    -   `GENERATED ALWAYS AS IDENTITY`: The database forces auto-generation. If a user tries to manually insert an ID value, the engine rejects the command immediately, preserving sequence consistency.

### 2. Under the Hood: Sequence Objects
A sequence is a standalone, single-row system table that acts as a numeric counter. It operates independently of transactional rollbacks: if you begin a transaction, fetch number `5` from a sequence, and then roll back the transaction, the sequence remains at `5`. The number `5` is burned and skipped, guaranteeing that sequence generation never blocks concurrent transactions.

## Code Example: Managing Sequences

Let's look at how to declare, inspect, and reset auto-incrementing columns.

### Defining Schema with IDENTITY
```sql
CREATE TABLE tasks (
    task_id INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    description VARCHAR(100) NOT NULL
);
```

When this table is created, PostgreSQL automatically generates a system sequence, typically named `tasks_task_id_seq`.

---

### Scenario A: Out of Sync Simulation
Suppose we manually insert rows with explicit IDs (e.g., copying historical data):

```sql
-- Explicitly insert task 1 and 2
INSERT INTO tasks (task_id, description) VALUES (1, 'Historical Task A');
INSERT INTO tasks (task_id, description) VALUES (2, 'Historical Task B');

-- At this point, our table has IDs 1 and 2. 
-- However, the underlying sequence generator is still at starting position 1!
```

If we try to insert a new row letting the database auto-generate the ID:

```sql
INSERT INTO tasks (description) VALUES ('New Auto Task');
-- Result: ERROR: duplicate key value violates unique constraint "tasks_pkey"
-- Detail: Key (task_id)=(1) already exists.
```
Why did it fail? Because the sequence generated `1`, which Alice is already using.

---

### Scenario B: Inspecting and Resetting the Sequence
To fix this, we inspect the maximum ID in the table and reset the sequence to restart *after* that number.

```sql
-- Step 1: Find the max ID
SELECT MAX(task_id) FROM tasks; -- Returns 2

-- Step 2: Reset the sequence to start at 3
ALTER TABLE tasks ALTER COLUMN task_id RESTART WITH 3;

-- Step 3: Test insertion again
INSERT INTO tasks (description) VALUES ('New Auto Task');
-- Result: SUCCESS (task_id 3 is successfully generated)
```

---

## Summary
-   **`SERIAL`** is legacy PostgreSQL syntax; **`GENERATED AS IDENTITY`** is the modern SQL-compliant standard.
-   Auto-incrementing columns are backed by **Sequence** objects, which generate values independently of transactions.
-   Manually inserting explicit IDs can cause sequences to fall **out of sync** with table records, leading to key duplicate errors.
-   Correct sequence offsets using **`ALTER TABLE ... RESTART WITH`**.
