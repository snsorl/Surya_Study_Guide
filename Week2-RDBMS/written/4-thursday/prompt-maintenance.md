# Prompt Maintenance and Versioning

## Learning Objectives
- Define Prompt Maintenance and explain its importance in preserving software quality.
- Implement version control principles (semantic versioning, changelogs) for prompt libraries.
- Analyze stored prompts to identify and remove stale, redundant, or misleading instructions.
- Construct a modular, reusable prompt catalog for development teams.

---

## Why This Matters
When you write Java code, you use Git to version your files. You track changes, check diffs, and write commit messages to explain why a method changed. However, when developers use AI tools to generate SQL schemas, test cases, or API endpoints, they often treat their prompts as temporary text blocks. 

They write a prompt, modify it in a chat window until it works, and throw it away.

In enterprise software engineering, this is a major operational risk. If an AI prompt is used to auto-generate database structures or parse customer profiles, **that prompt is source code**. If the prompt is not versioned, tested, and maintained, changes in the underlying LLM's behavioral model can cause the prompt to break, introducing logical bugs into your application.

Learning how to audit, version, and clean your prompt library guarantees that your AI-assisted pipelines remain predictable and secure.

---

## The Concept

### 1. What is Prompt Maintenance?
**Prompt Maintenance** is the continuous lifecycle practice of auditing, modifying, and versioning a library of prompt templates to ensure they remain compatible with updated models and software requirements.

### 2. Prompt Versioning (Infrastructure-as-Code)
Just like software versioning, prompts should follow a structured lifecycle:
-   **Semantic Versioning:** Major prompt changes (like switching from returning XML to JSON) require a major version bump (e.g., `v1.0.0` -> `v2.0.0`). Minor adjustments (like adding a validation constraint) require a minor bump (e.g., `v1.0.0` -> `v1.1.0`).
-   **Changelogs:** Every prompt template file should contain a header section documenting *when* it was changed, *who* changed it, and *what* behavior was optimized.

### 3. Pruning Stale and Misleading Prompts
As LLMs evolve, they become better at understanding short instructions, or they deprecate support for certain formatting tricks.
-   **Stale Prompts:** Instructions written for older models (like GPT-3) containing excessive formatting guidelines that confuse modern models (like Gemini 3.5).
-   **Misleading Prompts:** Prompts that use outdated codebase patterns. For example, if your team switches from `SERIAL` to `IDENTITY` primary keys, any prompt templates still instructing the AI to use `SERIAL` must be pruned or updated immediately to prevent code drift.

## Code Example: Prompt Version Control Document

Below is a prompt catalog entry illustrating version control, changelog maintenance, and instruction refinements for a SQL generation prompt.

```markdown
# Prompt Catalog: SQL-Create-Junction-Table
ID: PR-SQL-004
Domain: Persistence Design

## Metadata
- Latest Version: v2.1.0
- Owner: Data Engineering Team
- Target Model: Gemini 3.5 Flash

---

## Version History

### [v2.1.0] - 2026-07-12 (Current Version)
- **Changes:** Added constraints requiring standard ANSI 'GENERATED AS IDENTITY' and explicitly banned PostgreSQL legacy 'SERIAL' types to prevent schema migration incompatibilities.
- **Prompt Template:**
  ```
  [System]
  You are an expert PostgreSQL Database Engineer. Generate a DDL script for a junction table based on the user's input.
  
  [Constraints]
  1. Enforce referential integrity using named foreign keys.
  2. Use composite primary keys.
  3. DO NOT use the legacy 'SERIAL' type. Use 'GENERATED BY DEFAULT AS IDENTITY' for integer identifiers.
  4. Return ONLY the raw SQL code. No markdown wrappers.
  5. Do not use any emojis in your response.
  ```

---

### [v2.0.0] - 2026-05-18 (Obsolete)
- **Changes:** Upgraded prompt to enforce JSON compliance check.
- **Status:** **Obsolete**. Replaced by v2.1.0 due to legacy 'SERIAL' type generation bugs.

---

### [v1.0.0] - 2026-02-11 (Stale - Pruned)
- **Changes:** Original basic creation prompt.
- **Status:** **Pruned**. Omitted from active catalog because it contained extensive system descriptions that are redundant in modern models.
```

## Summary
-   Prompts used in production pipelines are **source code** and must be versioned and maintained.
-   Apply **Semantic Versioning** and maintain a **Changelog** for prompt libraries.
-   **Prune stale prompts** that use deprecated codebase patterns (like legacy SQL types) to prevent code drift.
-   Centralize prompts in structured catalogs so development teams share the same templates.
