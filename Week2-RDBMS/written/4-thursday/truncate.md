# TRUNCATE vs. DELETE

## Learning Objectives
- Differentiate between the SQL `TRUNCATE` and `DELETE` commands.
- Compare execution speeds, transaction rollback capabilities, and database locking behaviors of both commands.
- Explain how both commands handle auto-incrementing identity sequences.
- Evaluate scenarios to select the appropriate command for data clearing tasks.

---

## Why This Matters
As your application processes transactions, databases grow. Over time, you will need to clear out tables—whether removing expired session tokens, clearing temporary upload files, or resetting test data in staging databases. 

A common beginner mistake is to run `DELETE FROM table_name;` to clear large tables. While this works, it can be extremely slow and can crash or freeze your database server.

In enterprise software, selecting the wrong command to clear data directly impacts system uptime. `DELETE` and `TRUNCATE` may achieve similar end states (an empty table), but under the hood, they operate completely differently. Knowing when to use each prevents memory leaks, locking issues, and performance bottlenecks.

---

## The Concept

### 1. Underlying Command Mechanics
-   **`DELETE` (DML Operation):** Row-by-row deletion. The database engine inspects each row individually, checks for foreign key constraints, fires active database triggers, deletes the row, and records the delete event in the transaction log for rollback purposes.
-   **`TRUNCATE` (DDL Operation):** Page-level deallocation. Instead of scanning rows, `TRUNCATE` deallocates the data pages (the actual storage space on disk) that the table occupies. It is equivalent to dropping the table and recreating it immediately.

### 2. Key Differences Comparison

#### A. Execution Speed and Logging
-   `DELETE` writes every row deletion to the transaction log, making it slow on large datasets (millions of rows can take minutes).
-   `TRUNCATE` only logs the page deallocations. It executes almost instantly, regardless of table size.

#### B. Transaction and Rollback Behavior
-   In most modern RDBMS engines (including PostgreSQL), **both** `DELETE` and `TRUNCATE` can be executed within a transaction block and rolled back if needed.
-   *Note:* In older or alternative databases (like MySQL), `TRUNCATE` triggers an implicit commit and cannot be rolled back.

#### C. Identity Sequence Reset
-   `DELETE` does **not** reset auto-incrementing identity sequences (e.g., if you delete a user with ID 10 and insert a new user, their ID will be 11, not 1).
-   `TRUNCATE` typically **resets** the identity sequences back to 1. In PostgreSQL, you can control this using the `RESTART IDENTITY` or `CONTINUE IDENTITY` options.

#### D. Database Locks and Triggers
-   `DELETE` locks individual rows (Row-Level Lock) and triggers any defined `BEFORE/AFTER DELETE` database triggers.
-   `TRUNCATE` locks the entire table (Table-Level Lock), preventing any reads or writes during execution, and **does not** fire database triggers.

## Code Example: Sequence Reset Comparison

Let's write a SQL script illustrating how both commands affect an auto-incrementing sequence.

### Setup Table
```sql
CREATE TABLE transaction_logs (
    log_id INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    message VARCHAR(100) NOT NULL
);

INSERT INTO transaction_logs (message) VALUES ('Log A'), ('Log B'), ('Log C');
-- Current log_id sequence is at 3
```

---

### Scenario A: Using DELETE
```sql
DELETE FROM transaction_logs;

-- Insert a new log
INSERT INTO transaction_logs (message) VALUES ('Log D');
SELECT * FROM transaction_logs;
```
*Output:*
```
log_id | message
-------|---------
4      | Log D
```
Notice that the ID sequence **continued** at 4. The previous IDs (1-3) are skipped.

---

### Scenario B: Using TRUNCATE
Let's rebuild the table with 3 logs, and then truncate it:

```sql
-- Reset data to 3 logs
-- ...

TRUNCATE TABLE transaction_logs RESTART IDENTITY;

-- Insert a new log
INSERT INTO transaction_logs (message) VALUES ('Log E');
SELECT * FROM transaction_logs;
```
*Output:*
```
log_id | message
-------|---------
1      | Log E
```
Notice that the ID sequence **reset** back to 1.

---

## Summary
-   **`DELETE`** is a row-by-row DML command. It is slow, triggers deletes, and does not reset identity sequences.
-   **`TRUNCATE`** is a page-deallocating DDL command. It is fast, locks the table, bypasses triggers, and resets identity sequences.
-   In PostgreSQL, both commands support transaction rollbacks.
-   Use **`DELETE`** when filtering rows with `WHERE`; use **`TRUNCATE`** when clearing entire large datasets or resetting environments.
