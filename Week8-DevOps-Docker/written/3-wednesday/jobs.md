# GitLab CI/CD Jobs: Syntax, Artifacts, Caching, and Rules

## Learning Objectives
- Write GitLab CI/CD jobs using correct YAML syntax.
- Configure job parameters: `stage`, `image`, `script`, `artifacts`, and `cache`.
- Optimize pipeline dependency execution speed using the `needs` keyword.
- Apply conditional logic to jobs using `rules` and `when` constraints.

---

## Why This Matters
Continuous Integration pipelines are built from individual units of work called **jobs**. A job represents a specific task, such as compiling Java classes, running unit tests, or building a Docker image.

If your jobs are configured poorly, your pipeline will run slowly, consume excessive compute resources, and fail to pass outputs between stages. Mastering job configuration—including artifact sharing, compiler caching, and conditional execution—is key to building high-performing development pipelines.

---

## The Concept

### 1. Job Structure in GitLab CI
A GitLab CI pipeline is defined in a `.gitlab-ci.yml` file in your repository root. A job consists of a unique name and a set of instructions:

```yaml
compile-job:         # Job Name
  stage: build       # Stage group
  image: maven:latest # Runner environment image
  script:            # Commands to run
    - mvn compile
```

---

### 2. Job Parameters Explained

#### Stage
Associates the job with a specific phase in the pipeline (e.g., `build`, `test`, `deploy`). Jobs in the same stage run in parallel, while jobs in subsequent stages wait until previous stages complete.

#### Image
Specifies the Docker container image the GitLab Runner should pull to execute the job's script commands.

#### Artifacts
A list of files or directories generated by the job that should be saved and shared with downstream jobs or downloaded from the GitLab web interface. For example, compiling a Spring Boot application generates a `.jar` file that downstream deployment jobs need.

#### Cache
Used to persist temporary files (like downloaded Maven `.m2` dependencies or npm `node_modules` folders) across multiple pipeline runs. This avoids downloading the same dependencies repeatedly, speeding up build times.

---

### 3. Optimizing Speed: The `needs` Keyword
By default, GitLab jobs wait for the *entire* previous stage to complete before starting. This can create bottlenecks:

```
Default Stage Execution (Slow):
Stage: Build   [compile-jar] -----\
               [compile-docs] -----+===> Stage: Test [run-tests] (Waits for both)

Needs Directed Acyclic Graph (Fast):
[compile-jar] ===(needs)===> [run-tests] (Starts immediately when jar is ready, ignoring compile-docs)
```

By adding the **`needs`** keyword, you can bypass the stage order. A job will start as soon as the specific jobs listed in its `needs` section complete, regardless of other jobs running in the previous stage.

---

### 4. Conditional Logic: `rules` and `when`
You do not want every job to run on every commit. E.g., you only want to deploy to production when code is merged to the `main` branch.
- **`when`**: Configures execution triggers (e.g., `when: manual` for human gates, `when: on_failure` to run cleanup on error).
- **`rules`**: A list of conditions evaluated sequentially. If met, they configure the job to run or be skipped (e.g., matching git branches or checks for changed files).

## Code Examples and Walkthroughs

### 1. Robust Job Configuration with Caching and Rules
This example shows a GitLab CI job that compiles a Java application, caches dependencies, passes the compiled JAR as an artifact, and only runs on Merge Requests or commits to the `main` branch:

```yaml
# Define pipeline stages
stages:
  - build
  - test

# Job: Compile the Java application
compile-backend:
  stage: build
  image: maven:3.8-openjdk-17-slim
  
  # Cache Maven dependencies to speed up subsequent runs
  cache:
    key: maven-cache-key
    paths:
      - .m2/repository

  # Script commands to run in the container
  script:
    - mvn clean package -DskipTests -Dmaven.repo.local=.m2/repository

  # Keep the compiled JAR file for 1 week
  artifacts:
    name: "backend-build-${CI_COMMIT_REF_SLUG}"
    paths:
      - target/*.jar
    expire_in: 1 week

  # Run this job only for merges to main or merge request events
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
```

---

## Summary
- **GitLab CI/CD Jobs** are the building blocks of automated pipelines, configured in `.gitlab-ci.yml`.
- **Artifacts** pass compiled files to downstream stages, while **Caching** saves dependencies across pipeline runs to speed up builds.
- The **`needs`** keyword creates non-blocking dependency paths between specific jobs.
- **`rules`** and **`when`** properties apply conditional logic to control when jobs are executed or skipped.
