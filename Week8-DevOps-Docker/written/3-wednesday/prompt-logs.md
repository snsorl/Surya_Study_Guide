# Prompt Logs: Engineering and Evaluating AI-Generated Pipelines

## Learning Objectives
- Design effective AI prompts to generate syntactically correct `.gitlab-ci.yml` pipeline files.
- Evaluate AI-generated pipeline configurations for syntax validity, security risks, and caching performance.
- Apply template patterns to structure prompts for containerization and testing jobs.
- Document and archive AI prompts to build a reusable team prompt library.

---

## Why This Matters
Using AI assistants to write CI/CD pipeline code can speed up development. However, writing pipeline files is complex: a single incorrect space, deprecated property, or insecure credential pattern will cause the pipeline execution to fail.

If you paste simple prompts like "write a gitlab-ci.yml file," the AI will likely return generic configurations that lack critical caching setups, use deprecated syntax, or expose sensitive API keys. Learning how to write structured prompts and verify the AI's output is key to generating reliable pipeline code.

---

## The Concept

### 1. The Structure of a High-Quality Pipeline Prompt
To generate accurate pipeline configurations, your prompts must include the following context:
- **Language and Build Framework**: Detail the specific version of Java, Maven, npm, or Gradle you are using.
- **Pipeline Stages**: Specify the exact stages you want (e.g., `build`, `test`, `deploy`).
- **Caching Strategy**: Instruct the AI to configure local caches for dependencies (like the Maven `.m2` folder) to speed up build times.
- **Security Constraints**: Direct the AI to use environment variables for sensitive access keys and to avoid hardcoded credentials.

---

### 2. Evaluating AI-Generated Pipeline Output
Before applying an AI-generated configuration to your repository, verify the output against these checks:
- **Syntax Check**: Does the YAML use spaces instead of tabs? Are indentation levels correct?
- **Deprecated Fields**: Check for deprecated keywords (e.g., use `rules` instead of `only/except` blocks in newer GitLab versions).
- **Security Audit**: Are there any hardcoded passwords or API keys?
- **Caching Validation**: Is the cache key configured correctly? (E.g., using `${CI_COMMIT_REF_SLUG}` to isolate caches by branch).

## Code Examples and Walkthroughs

### 1. Template Prompt for Java Spring Boot Pipeline Generation
Below is an example of a structured prompt template you can use to generate a `.gitlab-ci.yml` file for a Java project:

```text
[CONTEXT]
Generate a .gitlab-ci.yml file for a Spring Boot 3.x web application built using Java 17 and Maven.

[STAGES]
The pipeline must define three stages: build, test, and deploy.

[JOB REQUIREMENTS]
1. Build Job: Use 'maven:3.9-eclipse-temurin-17' as the image. Run 'mvn compile'. Configure local dependency caching for '.m2/repository' using a cache key based on the active branch name.
2. Test Job: Run 'mvn test'. Save the test reports in 'target/surefire-reports/' as an artifact for 3 days.
3. Deploy Job: Target AWS ECS. Use the 'amazon/aws-cli:latest' image. Run the command 'aws ecs update-service' referencing the cluster 'project3-cluster' and service 'api-service'. This job must run only on the 'main' branch and require manual approval to execute.

[SECURITY]
Ensure the AWS region is configurable via a default variable called 'AWS_DEFAULT_REGION' set to 'us-east-1'. Do not write hardcoded credentials in the script.
```

### 2. Evaluating and Fixing an Insecure AI Output

#### Insecure AI-Generated Output (Vulnerable to credential leaks)
Below is a poorly generated job where the AI hardcoded secret keys directly in the script block:

```yaml
# INSECURE: Generated by generic AI prompt
deploy-job:
  stage: deploy
  image: amazon/aws-cli:latest
  script:
    # CRITICAL VULNERABILITY: Credentials exposed in code repository!
    - export AWS_ACCESS_KEY_ID="AKIASECRETACCESSKEY"
    - export AWS_SECRET_ACCESS_KEY="SuperSecretKeyExample"
    - aws ecs update-service --cluster prod-cluster --service api
```

#### Remediation (Secure variable reference)
Refactor the job to reference protected environment variables configured in GitLab settings:

```yaml
# SECURE: Refactored configuration
deploy-job:
  stage: deploy
  image: amazon/aws-cli:latest
  script:
    # AWS CLI resolves these environment variables from GitLab settings automatically
    - aws ecs update-service --cluster prod-cluster --service api
```

---

## Summary
- **Structured Prompts** must include the build framework, pipeline stages, caching requirements, and security rules to generate accurate YAML configurations.
- Always review AI outputs for **syntax validity**, **deprecated properties**, and **security leaks**.
- Set **`rules`** instead of legacy `only/except` blocks, and leverage **GitLab environment variables** to keep credentials secure.
