# Analyzing AI-Generated Code for Security Vulnerabilities

## Learning Objectives
- Identify common security vulnerabilities in AI-generated code.
- Analyze vulnerable output patterns (SQL injection, hardcoded secrets, directory traversal).
- Execute a structured code review process on AI-generated snippets.
- Refactor insecure AI code into secure, production-ready solutions.

---

## Why This Matters
Generative AI models excel at producing syntactically correct, functional code quickly. However, AI models do not understand context, business risks, or security boundaries. They often generate code that works under ideal conditions but breaks catastrophically under attack.

Common issues include using string formatting instead of parameterized SQL queries, using weak cryptographic functions, or omitting input validation. As an engineer, you must treat all AI-generated code as **untrusted user input**. You must analyze and review every line of code generated by an AI before it is committed to production.

---

## Common Insecure AI Output Patterns

### 1. SQL Injection via String Concatenation
AI models frequently generate queries using string interpolation because it is simpler and more concise than using parameterized query interfaces.

```python
# --- INSECURE AI OUTPUT ---
def get_user(username):
    # DANGEROUS: Direct string insertion allows SQL injection
    query = f"SELECT * FROM users WHERE username = '{username}'"
    return db.execute(query)
```

### 2. Hardcoded Credentials and Secrets
When asked to write authentication logic or database connectors, AI models frequently generate placeholder credentials or hardcoded keys directly in the script.

```python
# --- INSECURE AI OUTPUT ---
def connect_to_service():
    # DANGEROUS: Exposes secrets in version control
    api_key = "d3b07384d113edec49eaa6238ad5ff00"
    return Client(api_key=api_key)
```

### 3. Missing Input Validation and Path Traversal
When asked to write file-serving handlers, AI models often forget to sanitize incoming filenames, opening the application to path traversal attacks (allowing users to read `/etc/passwd` or system logs).

```python
# --- INSECURE AI OUTPUT ---
@app.route("/files/<filename>")
def serve_file(filename):
    # DANGEROUS: Allows directory traversal (e.g., filename="../../etc/passwd")
    return send_from_directory("uploads", filename)
```

## Code Examples

### 1. Analyzing and Fixing a Vulnerable File Reader

#### The AI-Generated Snippet (Vulnerable):
```python
import os
from flask import Flask, request, jsonify

app = Flask(__name__)

@app.route("/read-profile", methods=["GET"])
def read_profile():
    # AI generated this without verifying filename bounds
    filename = request.args.get("file")
    filepath = os.path.join("user_profiles", filename)
    
    with open(filepath, "r") as f:
        content = f.read()
    return jsonify({"data": content}), 200
```
*Vulnerability:* An attacker could send a request like `/read-profile?file=../../etc/passwd` or `/read-profile?file=..\..\windows\win.ini`. The `os.path.join` helper resolves relative directory paths, allowing the attacker to read arbitrary system files.

---

#### The Secured Refactored Snippet (Fixed):
To secure the file-serving endpoint, we must sanitize the filename using secure library utils and check that the resolved path resides strictly within the intended target directory:

```python
import os
from pathlib import Path
from flask import Flask, request, jsonify, abort
from werkzeug.utils import secure_filename

app = Flask(__name__)
# Resolve absolute base directory path
UPLOAD_DIR = Path.cwd() / "user_profiles"

@app.route("/read-profile", methods=["GET"])
def read_profile():
    filename = request.args.get("file")
    if not filename:
        return jsonify({"error": "File parameter is required"}), 400

    # 1. Clean the filename to remove paths (e.g., '../../passwd' -> 'passwd')
    safe_name = secure_filename(filename)
    
    # 2. Resolve absolute path
    resolved_path = (UPLOAD_DIR / safe_name).resolve()
    
    # 3. Security Boundary Check: Ensure path is within UPLOAD_DIR
    if not resolved_path.is_relative_to(UPLOAD_DIR):
        abort(403)  # Forbidden access attempt
        
    try:
        with open(resolved_path, "r") as f:
            content = f.read()
        return jsonify({"data": content}), 200
    except FileNotFoundError:
        return jsonify({"error": "Profile file not found"}), 404
```

## AI Code Review Checklist

When reviewing any code block generated by an AI assistant, verify the following checks:

- [ ] **SQL Queries:** Are all SQL statements parameterized using place markers (`?` or `%s`), completely avoiding string interpolation (`f""` or `+`)?
- [ ] **Secrets:** Are there any hardcoded passwords, tokens, private keys, or API credentials? (All should be loaded from `os.environ` or a secrets manager).
- [ ] **Input Sanitization:** Is user-provided data checked for length, bounds, character type, and path segments before being used in system operations?
- [ ] **File Operations:** Are file paths resolved using `Path().resolve()` and verified against base directories to prevent path traversal?
- [ ] **Cryptographic Primitives:** Are strong, modern hashing algorithms used (like `bcrypt` or `argon2` for passwords) instead of deprecated MD5/SHA1 algorithms?

---

## Summary
- AI-generated code is inherently untrusted and frequently contains subtle, severe security vulnerabilities.
- Look out for SQL injection via string manipulation, hardcoded credentials, and path traversal in file paths.
- Secure file operations by sanitizing names with `secure_filename()` and verifying boundaries using `Path().is_relative_to()`.
- Implement a strict AI Code Review Checklist before committing any AI-assisted code.
