# Compiled vs. Interpreted: Java vs. Python Execution Models

## Learning Objectives
- Compare and contrast compiled languages with interpreted languages.
- Detail the multi-stage compilation and execution process of Java.
- Describe how CPython compiles source code to bytecode and executes it line-by-line.
- Utilize Python's REPL for interactive code execution and experimentation.
- Explain the role of `.pyc` files and `__pycache__` directories.
- Define Just-In-Time (JIT) compilation and its application in PyPy.

---

## Why This Matters
As you start writing Python, you will notice that you do not need to run a compile command (like `javac`) before running your code. You simply run `python script.py`, and it runs instantly.

To write high-performance applications and diagnose deployment bugs, you must understand what happens under the hood when a computer processes your code. Understanding the execution difference between Java's virtual machine (JVM) and Python's interpreter runtime (CPython) will help you make better design decisions and optimize performance in polyglot microservice environments.

---

## The Concept

### 1. Compiled vs. Interpreted Execution Models
In a purely **compiled** language (like C or Go), code is translated into machine-specific assembly instructions *before* execution. 

Java and Python are hybrid languages, but they approach the execution boundary differently:

#### Java: Ahead-of-Time Compiling to JVM Bytecode
When you run `javac MyClass.java`, the compiler translates your source code into an intermediate format called **Java Bytecode** (saved in `.class` files). This bytecode is platform-independent. To execute it, the **Java Virtual Machine (JVM)** interprets the bytecode, and uses its own JIT compiler to compile hot paths into raw machine instructions for execution.

```
[Java Source (.java)] --> (javac Compiler) --> [Bytecode (.class)] --> (JVM Interpreter & JIT) --> [Machine Code]
```

#### Python: Interpreted Line-by-Line Execution
Python is classified as an **interpreted** language because CPython (the default implementation of Python written in C) performs compilation automatically and transparently at runtime. 
1.  When you run a script, Python compiles it into **Python Bytecode** in memory.
2.  It then feeds this bytecode into the **CPython Virtual Machine**, which executes it line-by-line.

```
[Python Source (.py)] --> (CPython Parser/Compiler) --> [Python Bytecode (.pyc)] --> (CPython VM Interpreter) --> [Execution]
```

### 2. The REPL (Read-Eval-Print Loop)
Because Python executes instructions dynamically, it supports a **REPL (Read-Eval-Print Loop)** interface. The REPL is an interactive command-line environment where you can type code line-by-line and see immediate execution results.
- **Read:** Reads user input code.
- **Eval:** Evaluates the code in the active namespace.
- **Print:** Prints the return value of the evaluation.
- **Loop:** Repeats the process.

This is highly useful for prototyping APIs, testing syntax, and inspecting data frames.

### 3. Python Bytecode & `__pycache__`
Although Python is interpreted, it does cache compiled bytecode to save compilation time on future runs. 
- When Python imports a module, it compiles the source `.py` file to bytecode `.pyc` (Python Compiled) and stores it in a directory named `__pycache__`.
- On subsequent runs, if the source code has not changed, Python skips compilation and directly loads the cached `.pyc` file, speeding up application startup.

### 4. PyPy and JIT Compilation
The standard CPython runtime does not perform **Just-In-Time (JIT)** compilation of bytecode to machine code. This means every instruction goes through the interpreter, which is slower than compiled Java.
- **PyPy** is an alternative Python interpreter written in Python itself.
- PyPy features a **JIT compiler**. As the program runs, PyPy identifies "hot loops" (code blocks run repeatedly) and compiles them directly into native machine code.
- PyPy can execute CPU-bound Python code up to 5x to 10x faster than standard CPython.

## Code Examples

### 1. Using the Python REPL
You can start the REPL by typing `python` (or `python3` on macOS/Linux) in your terminal. Here is a sample session:

```python
>>> # Read, Eval, Print, Loop in action
>>> x = 10
>>> y = 20
>>> x + y
30
>>> import math
>>> math.sqrt(x + y)
5.477225575051661
```

### 2. Viewing Python Bytecode
We can use Python's built-in `dis` (disassembler) module to inspect the actual bytecode generated by the interpreter for a function. This is equivalent to using `javap -c` on a Java class file.

```python
import dis

def calculate_bonus(salary):
    return salary * 0.15

# Disassemble the function into Python Bytecode instructions
dis.dis(calculate_bonus)
```

**Bytecode Output:**
When run, Python prints the instructions the CPython VM executes:
```text
  4           0 RESUME                   0

  5           2 LOAD_FAST                0 (salary)
              4 LOAD_CONST               1 (0.15)
              6 BINARY_OP                5 (*)
             10 RETURN_VALUE
```
- `LOAD_FAST`: Loads local variable `salary` onto the evaluation stack.
- `LOAD_CONST`: Loads constant float value `0.15` onto the stack.
- `BINARY_OP`: Multiplies (`*`) the two top values on the stack.
- `RETURN_VALUE`: Returns the product value from the function.

---

## Summary
- **Java** requires explicit compilation (using `javac`) to build bytecode `.class` files, which are run on a JVM.
- **Python** uses CPython to automatically parse and compile code to bytecode in-memory, running it line-by-line through the interpreter.
- **REPL** enables immediate feedback for testing Python syntax.
- **`__pycache__`** holds cached `.pyc` files of imported modules to improve startup performance.
- **PyPy** provides a JIT compiler that optimizes Python execution by converting hot paths directly into native machine code.
